## Summary

Data engineers can extract incremental changes from any `_log` table or a point-in-time
snapshot of any base table via a secure, paginated REST API. Extraction windows are tracked
per user per table in `data_extraction_execution`, ensuring exactly-once delivery with
idempotent retries. The feature is Enterprise-tier only. See [research.md](research.md),
[data-model.md](data-model.md), [contracts/api.md](contracts/api.md), and
[quickstart.md](quickstart.md).

## Technical Context

**Language/Version**: Go 1.26+ (API), TypeScript / SvelteKit 5 (frontend)  
**Primary Dependencies**: chi router, pgx/v5, svelte-i18n, Tailwind CSS v4  
**Storage**: PostgreSQL 18+, `wd` schema, UUIDv7 PKs, raw SQL migrations via Goose  
**Testing**: `go test -race ./...` (integration: testcontainers), Playwright (e2e)  
**Target Platform**: Linux server (API), web browser (frontend)  
**Project Type**: Web service + SPA  
**Performance Goals**: Current extraction of 100k rows completes in < 60 seconds  
**Constraints**: Row count cap 10,000 per page; safety lag default 5s  
**Scale/Scope**: Per-tenant; multiple concurrent data engineers supported

## Phase 0 Artifacts

All research complete. See [research.md](research.md).

Key decisions:
- Existing `dataexport` handler replaced entirely (prototype reads `audit_log`; spec reads `_log` tables)
- `data_engineer` is both API key scope (programmatic) and user role (UI session)
- Enterprise tier gated via `data_api_enabled` on `platform_usage_tier`
- Discovery via `information_schema` — schema-driven, requires no code changes for new tables
- Partial unique index enforces idempotent page 1 retry at DB level
